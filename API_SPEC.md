# Спецификация REST API: Система «АвтоСклад» (Нормализованная архитектура)

**Базовый URL:** `/api/v1`
**Формат данных:** `application/json`
**Авторизация:** Bearer Token (JWT) в заголовке `Authorization: Bearer <token>` *(опционально)*

---

## 1. Справочники классификаторов (Brands & Locations)
Выделенные сущности для обеспечения целостности данных (нормализация).

### 1.1 Бренды (Brands)
`GET /brands` — получить список всех брендов.
`POST /brands` — создать новый бренд.
```json
// POST /brands
{
  "name": "VAG"
}
// Ответ: { "id": "b1", "name": "VAG" }
```

### 1.2 Места хранения (Locations)
`GET /locations` — получить список всех мест хранения (полок/стеллажей).
`POST /locations` — создать новое место хранения.
```json
// POST /locations
{
  "name": "Стеллаж А-12"
}
// Ответ: { "id": "loc1", "name": "Стеллаж А-12" }
```

## 2. Справочник Номенклатуры (Catalog)
Хранит **только статические данные** товаров. Связан с брендами и местами хранения через внешние ключи (Foreign Keys).

### 2.1 Получить список номенклатуры
`GET /catalog`

**Query-параметры:**
* `search` (string) — поиск по артикулу или названию.
* `type` (string) — фильтр по типу: `real` (реальные) или `phantom` (фантомы/кроссы).
* `parentId` (string) — получить все фантомы конкретного товара.

**Успешный ответ (200 OK):**
```json
{
  "data": [
    {
      "id": "uuid-1",
      "article": "L06L109259E",
      "brandId": "b1",
      "name": "Магнит клапана фазорегулятора",
      "locationId": "loc1",
      "comment": "Ходовой товар",
      "type": "real",
      "parentId": null
    },
    {
      "id": "uuid-2",
      "article": "06L109259a",
      "brandId": "b1",
      "name": "Магнит клапана (кросс)",
      "locationId": null,
      "comment": null,
      "type": "phantom",
      "parentId": "uuid-1"
    }
  ],
  "total": 2
}
```

### 2.2 Создать товар или фантом
`POST /catalog`

**Тело запроса:**
```json
{
  "article": "L06L109259E",
  "brandId": "b1",
  "name": "Магнит клапана",
  "locationId": "loc1",
  "type": "real" // или "phantom" с указанием "parentId"
}
```

### 2.3 Обновить карточку товара
`PATCH /catalog/:id`
Обновляет `locationId`, `comment` или другие текстовые поля.

## 3. Регистр Остатков (Stock)
Связывает ID товара с текущим физическим наличием на складе.

### 3.1 Получить текущие остатки
`GET /stock`

**Успешный ответ:**
```json
{
  "data": [
    {
      "productId": "uuid-1",
      "qty": 15
    }
  ]
}
```
*(Примечание: Остаток фантомов равен остатку их `parentId`, бэкенд может возвращать их вычисляемо, либо фронтенд мапит их сам).*

---

## 4. Регистр Цен (Pricing)
Отдельная сущность для хранения закупочных и продажных цен.

### 4.1 Получить цены
`GET /prices`

**Успешный ответ:**
```json
{
  "data": [
    {
      "productId": "uuid-1",
      "purchasePrice": 1200,
      "sellingPrice": 1500
    },
    {
      "productId": "uuid-2", // Фантом может иметь свою цену продажи
      "purchasePrice": null,
      "sellingPrice": 1400
    }
  ]
}
```

### 4.2 Установить/обновить цены вручную
`PUT /prices/:productId`

**Тело запроса:**
```json
{
  "purchasePrice": 1200,
  "sellingPrice": 1500
}
```

## 5. Справочник Контрагентов (Partners)
Единый справочник для поставщиков и покупателей.

### 5.1 Получить список контрагентов
`GET /partners`
**Query-параметры:** `type` (`supplier` | `client` | `all`)

### 5.2 Добавить контрагента
`POST /partners`
```json
{
  "name": "ООО Ромашка",
  "type": "supplier"
}
```

## 6. Документы и Движение товара (Documents)
Приход и расход оформляются документами. Проведение документа автоматически делает движения по регистрам (Остатки и Цены).

### 6.1 Получить список документов
`GET /documents`
**Query-параметры:** `type` (`income` | `expense`), `partnerId`, `startDate`, `endDate`

### 6.2 Создать и провести документ (Приход / Расход)
`POST /documents`

Создает документ и **автоматически обновляет регистры**.
*   **Income (Приход):** Увеличивает `qty` в `Stock` и обновляет `purchasePrice` в `Pricing`.
*   **Expense (Расход):** Уменьшает `qty` в `Stock` (с проверкой на отрицательные остатки).

**Тело запроса:**
```json
{
  "type": "expense",
  "date": "2026-03-20T10:00:00Z",
  "partnerId": "c1",
  "rows": [
    {
      "productId": "uuid-1",
      "qty": 2,
      "price": 1500
    }
  ],
  "totalAmount": 3000
}
```

### 6.3 Откат (удаление) документа
`POST /documents/:id/rollback` (или `DELETE /documents/:id`)

Отменяет проведение документа:
* Возвращает остатки на склад (для `expense` — прибавляет, для `income` — вычитает).
* Удаляет запись о документе из истории.

## 7. Сводные данные (Views / Reports)
Для удобства фронтенда, чтобы не делать сложные JOIN-ы на клиенте для таблиц.

### 7.1 Получить сводную карточку товаров (ProductView)
`GET /views/products`

Возвращает "склеенные" данные: Номенклатура + Бренд + Место + Остаток + Цены. Это основной эндпоинт для отображения таблиц на фронтенде.

**Query-параметры:**
* `search` (string)
* `availability` (`in` | `out` | `all`)

**Успешный ответ:**
```json
{
  "data": [
    {
      "id": "uuid-1",
      "article": "L06L109259E",
      "brandId": "b1",
      "brand": "VAG",
      "name": "Магнит клапана",
      "locationId": "loc1",
      "location": "Стеллаж А-12",
      "type": "real",
      "qty": 15,
      "purchasePrice": 1200,
      "sellingPrice": 1500
    }
  ]
}
```

---

### 💡 Рекомендации для Backend-разработчика:
1. **Нормализация БД:** База данных должна состоять из таблиц: `brands`, `locations`, `catalog`, `stock`, `prices`, `partners`, `documents`, `document_rows`.
2. **Транзакции БД (ACID):** Эндпоинты `POST /documents` и `POST /documents/:id/rollback` должны выполняться строго в транзакции. Запись/удаление шапки документа, строк документа, обновление остатков и цен должны либо примениться вместе, либо откатиться при ошибке (например, нехватке товара при расходе).
3. **Логика Фантомов (Кроссов):** Фантомы хранятся в таблице `catalog` с `type = 'phantom'` и заполненным `parentId`. При запросе остатков фантома (`GET /views/products`), бэкенд должен подтягивать `qty` от его `parentId`. При этом продажная цена (`sellingPrice`) у фантома может быть своя (берется из таблицы `prices` по ID фантома).
4. **Soft Delete:** Для справочников (товары, контрагенты, бренды, локации) используйте "мягкое удаление" (флаг `is_deleted = true`), чтобы не нарушать ссылочную целостность в уже проведенных исторических документах.
